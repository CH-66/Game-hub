# 用 Codex 完成「朋友之间网页在线对战的跳跳棋」项目指南

下面是一份可直接交给 Codex 的执行指南，包含项目拆分、技术路线、里程碑与可执行的交互方式。你可以复制这些段落到与 Codex 的对话中，逐步推进。

---

## 1. 先确定范围与目标

在开始前，请确认以下要点（可在对话中直接告诉 Codex）：

- 平台：网页（PC+移动端），浏览器即可进入
- 模式：朋友之间在线对战（1v1 或多人）
- 游戏：标准跳跳棋规则（你也可以指定变体）
- 结果：可部署的 Web 应用（含前端、后端、实时对战）

建议把下面这段话直接发给 Codex 作为项目目标：

> 我想做一个可在线对战的网页跳跳棋游戏，要求支持好友房间、实时同步、断线重连、基本防作弊。请拆成可交付里程碑，并逐步实现。

---

## 2. 推荐技术栈（Codex 会基于它交付）

如果你没有强偏好，推荐如下：

- 前端：React + Vite + TypeScript
- 后端：Node.js + WebSocket（或 Socket.IO）
- 数据：内存房间 + Redis（可选，用于断线重连与状态持久化）
- 部署：Cloudflare Pages（前端）+ Cloudflare Workers（后端）或任一云主机

你可以在一开始就告诉 Codex 使用该栈，避免反复确认。

---

## 3. 项目里程碑（建议按顺序让 Codex 实现）

### 里程碑 1：规则引擎与本地对战

- 实现棋盘建模、棋子合法走法、胜利判定
- 前端可单机对弈
- 输出基础 UI

### 里程碑 2：房间与在线对战

- 创建房间、加入房间、匹配好友
- WebSocket 实时同步
- 玩家状态（准备、断线、重连）

### 里程碑 3：防作弊与可回放

- 服务端权威状态
- 客户端仅发送意图
- 对局记录与回放

### 里程碑 4：上线部署

- 生产环境配置
- 部署文档与一键脚本

---

## 4. 如何和 Codex 高效协作（建议用法）

你可以用“任务分解 + 指令执行”的方式驱动 Codex。以下是可直接使用的模板：

**模板 A：让 Codex 生成里程碑**

> 请把项目拆成 4 个里程碑，每个里程碑给出文件结构、主要类/模块、接口和验收标准。

**模板 B：逐里程碑执行**

> 现在做里程碑 1，先搭项目骨架并实现棋盘渲染、落子与合法走法判断。完成后告诉我如何运行。

**模板 C：逐模块实现**

> 先实现服务端房间系统（创建、加入、离开、房主转移），给出接口约定与核心代码。

**模板 D：代码审查**

> 请检查是否存在同步与状态不一致问题，指出可能的竞态和修复建议。

---

## 5. 你需要提供给 Codex 的最小信息清单

下面的信息越明确，Codex 出活越快：

- 规则：标准跳跳棋还是自定义？棋盘大小？人数？
- 对战：是否允许观战、聊天、悔棋？
- UI：是否已有设计稿或参考链接？
- 部署：想部署到哪里？是否需要域名？

如果你没有这些信息，也可以告诉 Codex “先按默认实现”。

---

## 6. 交付与验收清单（对 Codex 的检查点）

你可以用下面的清单让 Codex 自检：

- 能本地运行（启动命令清晰）
- 房间创建/加入/断线重连可用
- 双方走子同步一致
- 规则完整（跳跃、多跳、胜负判定）
- 服务端权威状态，客户端无权修改结果
- 有部署说明

---

## 7. 一段完整可复制的开场指令

把下面这段直接发给 Codex 开始执行：

> 我想做一个朋友之间网页在线对战的跳跳棋游戏。请使用 React + Vite + TypeScript 的前端，Node.js + WebSocket 的后端。先给出 4 个里程碑与验收标准，然后从里程碑 1 开始实施。实施时请给出具体文件结构、代码、运行命令，并在每个里程碑结束时总结已完成内容和下一步。
